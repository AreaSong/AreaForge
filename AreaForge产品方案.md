# AreaForge 产品方案

## 1. 产品定位

**AreaForge** 是一个面向个人长期备考的自我锻造与考研督战系统。

它不是普通打卡软件，也不是单纯的待办清单，而是围绕一个长期目标持续追踪学习投入、任务完成、知识掌握、错题复盘、状态波动和阶段风险的私人作战台。

当前第一目标：

- 考试方向：考研计算机，公共课 + 408
- 目标院校：清华
- 目标分数：450
- 当前基础：零基础
- 起始日期：2026-07-05
- 使用设备：电脑为主，手机作为补充访问

## 2. 命名与气质

产品名：**AreaForge**

含义：

- `Area` 延续已有产品体系，如 AreaFlow、AreaMatrix。
- `Forge` 表示锻造、淬炼、把当前的自己打造成更强的自己。

中文气质关键词：

- 锻造
- 督战
- 复盘
- 压强
- 长期主义
- 冷静但不麻木

可选中文副标题：

- AreaForge 考研督战台
- AreaForge 自我锻造系统
- AreaForge 研途作战台

## 3. 核心原则

1. **先让用户每天回来**
   - 第一优先级不是功能多，而是每天能打开、能开始、能完成一次学习闭环。

2. **数据必须推动行动**
   - 不只是展示学习时长，而是根据时长、任务、进度和断签情况，给出下一步建议。

3. **鞭策必须和状态相关**
   - 文案、主题和压力等级应根据用户真实进度变化，而不是固定鸡汤或固定毒舌。

4. **狠但不摧毁**
   - 可以尖锐，可以冷静揭穿问题，但每次都要把用户推回一个可执行动作。

5. **数据和代码分离**
   - 学习记录、错题、复盘、计划等数据必须持久化保存，应用更新不能破坏数据。

## 4. 推荐形态

第一版推荐做成：**私有 Web 应用**。

原因：

- 用户主要在电脑端学习，Web 最适合电脑使用场景。
- 可以部署到自己的服务器，随时从电脑和手机访问。
- 不需要先做手机 App、小程序或应用商店发布。
- 后续可以升级为 PWA，让手机像 App 一样添加到桌面。

第一版部署目标：

- 私有访问
- 登录保护
- 数据存储在服务器数据库
- 支持后续自动部署
- 支持版本展示和更新日志

## 5. 核心模块

### 5.1 今日作战台

用于每天打开后的第一屏。

包含：

- 今日日期
- 距离目标考试日剩余天数
- 今日学习总时长
- 今日任务完成率
- 当前连续打卡天数
- 当前阶段名称
- 当前风险等级
- 今日鞭策文案
- 快速开始学习计时

### 5.2 任务计划

用于管理每天应该完成什么。

任务字段：

- 科目
- 任务标题
- 任务类型
- 预计时长
- 实际时长
- 优先级
- 完成状态
- 完成后的复盘记录

科目初始建议：

- 数学
- 英语
- 408 数据结构
- 408 计算机组成原理
- 408 操作系统
- 408 计算机网络
- 政治

### 5.3 学习计时

用于记录真实学习投入。

能力：

- 选择科目开始计时
- 暂停/继续
- 结束后填写学习质量
- 标记有效学习或低效学习
- 关联到任务

可记录信息：

- 科目
- 开始时间
- 结束时间
- 持续时长
- 学习质量评分
- 是否有效学习
- 学习备注

### 5.4 错题与知识点复盘

用于沉淀问题，而不是只追求学习时长。

记录字段：

- 科目
- 来源
- 题目或知识点标题
- 错误原因
- 正确思路
- 复习状态
- 下次复习日期

错误原因分类：

- 完全不会
- 概念混乱
- 公式不熟
- 思路错误
- 粗心
- 时间不够
- 题型不熟

### 5.5 每晚复盘

用于每天结束时形成闭环。

复盘问题：

- 今天学了多久？
- 今天完成了什么？
- 今天哪里失控了？
- 今天最该保留的一个动作是什么？
- 明天最小必须完成任务是什么？

### 5.6 鞭策引擎

根据真实进度动态生成反馈。

基础语气类型：

- 温和鼓励型
- 严肃教练型
- 毒舌逼迫型
- 数据冷冰冰型

状态类型：

- 上升期：连续学习、完成率高、进度健康
- 稳定期：持续学习，但增长一般
- 波动期：有学习，但经常拖延或质量不稳
- 失守期：连续多天未打卡或任务大量失败
- 危险期：长期落后，计划明显失效
- 冲刺期：临近考试，优先真题、模拟、错题复盘

原则：

- 学得好时，不灌鸡汤，继续加压。
- 学得差时，不只批评，必须给出最低可执行动作。
- 长期断签时，进入恢复模式，先恢复行动感。

### 5.7 动机档案

用于保存最私密的开始原因。

建议设计：

- 默认不在首页频繁展示。
- 只有在连续失守、深夜复盘失败、或用户主动查看时出现。
- 不把过去的人变成每天的刺激源，而是把那段经历变成底层燃料。

### 5.8 数据统计

用于长期追踪。

第一版统计：

- 每日学习时长
- 每周学习时长
- 科目投入占比
- 任务完成率
- 连续打卡天数
- 断签次数
- 错题数量
- 复盘完成率

后续统计：

- 阶段目标完成率
- 预测风险
- 科目短板提醒
- 按目标分倒推的投入建议

## 6. 阶段计划

当前需要先确认目标考试日。

如果目标是常见意义上的 27 考研：

- 初试时间大概率在 2026 年 12 月下旬
- 从 2026-07-05 开始，准备时间约 5 个多月
- 策略会非常紧张，需要压缩为抢救型备考计划

如果目标是 2027 年 12 月下旬初试：

- 更接近通常意义上的 28 考研
- 从 2026-07-05 开始，准备时间约 17 个多月
- 可以做长期基础、强化、真题、冲刺四阶段规划

暂定长期四阶段：

1. 2026-07 到 2026-12：基础唤醒期
2. 2027-01 到 2027-06：系统强化期
3. 2027-07 到 2027-09：真题专题期
4. 2027-10 到 2027-12：冲刺模拟期

## 7. 部署与更新方案

推荐部署方式：

- 私有 Web 应用
- Docker Compose 部署
- 数据库独立持久化
- GitHub Actions 自动构建
- 服务器自动拉取新版本并重启
- 应用启动时自动执行数据库迁移

关键目标：

- 更新代码不影响学习数据
- 不需要每次手动拉代码、重启、整合数据
- 后续可以在后台显示当前版本、最新版本和更新日志

第一阶段建议：

- 自动部署先通过 GitHub Actions 完成
- 管理后台只展示版本信息和更新日志
- 暂不做网页内“一键更新”，避免过早引入服务器命令执行权限风险

第二阶段可做：

- 后台检查 GitHub Release
- 显示新版本可用
- 查看更新日志
- 管理员手动确认更新
- 后台触发安全的部署流程
