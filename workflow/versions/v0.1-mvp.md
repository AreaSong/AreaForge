# v0.1 前置主闭环计划

## 目标

把 AreaForge 从工程骨架推进到“个人自用可跑通”的前置主闭环。

v0.1 不是 `docs/product/feature-scope.md` 中的完整第一版；它只证明最核心学习链路可以真实保存和刷新。完整第一版还需要后续 `0004`、`0005`、`0008` 到 `0011` 完成。

v0.1 必须能完成：

> 登录 -> 创建任务 -> 开始计时 -> 结束计时 -> 保存记录 -> 晚间复盘 -> 首页统计刷新

## 范围

- 单管理员登录。
- 初始化管理员和科目。
- 今日作战台读取真实数据。
- 每日任务 CRUD。
- 专注计时持久化。
- 打卡与连续天数。
- 晚间复盘保存。
- 基础统计。
- 本地规则鞭策文案。

## 可选但不阻塞 v0.1

- 考纲树基础版。
- 笔记基础版。
- AI 鞭策建议。

当前考纲树和笔记基础版已开始落地；附件上传与 AI 实际调用仍按高风险边界单独确认。

## 后续版本衔接

- `0004` 完成后进入 AI 或任务债务闭环，具体顺序以 `docs/development/implementation-order.md` 为准。
- `0004`、`0005`、`0008` 到 `0011` 覆盖 `docs/product/feature-scope.md` 的完整第一版必须项。
- `0012` 到 `0014` 覆盖第二阶段增强和部署发布闭环。
- 暂缓项仍不进入当前版本：多用户、原生 App、小程序、复杂权限、AI 自动完整计划、复杂 PDF 自动解析和网页内一键更新。

## 不包含

- 多用户系统。
- 原生 App。
- 小程序。
- 高级权限。
- AI 自动生成完整计划。
- PDF 自动解析。
- 网页内一键更新。

## 入口条件

- `docs/README.md`、`README.md`、`AGENTS.md` 源事实入口一致。
- `docs/development/pre-code-closure.md` 验收通过。
- `tasks/active` 有当前任务。
- `tasks/backlog` 有后续 MVP 任务。
- `tasks/done/0006-engineering-prep-before-mvp.md` 中的工程准备断点已处理，或有明确延期理由。
- `tasks/done/0007-lightweight-codex-governance.md` 中的轻量治理规范已完成，或确认不阻塞 v0.1。

## 验收标准

- 登录保护生效。
- 任务、计时和复盘都写入数据库。
- 首页统计来自数据库，不再依赖 mock 数据。
- 刷新页面后核心记录不丢失。
- `pnpm check` 通过。
- 高风险行为没有绕过确认或审计。

## 退出条件

- 用户可以连续 3 天用 AreaForge 完成前置每日闭环。
- 每天至少能看到任务、计时、复盘、统计四类真实数据。
- 没有阻塞自用的 P0 / P1 问题。

## 完整第一版出口

完整第一版必须继续满足 `docs/product/feature-scope.md` 的“第一版必须有”，包括任务债务、打卡、考纲进度树、掌握证明、笔记与资料上传、情绪与状态、恢复模式、反假学习、作战地图、动机封存、阶段称号、AI 复盘建议和明日任务建议。

## 风险

- 认证和会话属于高风险边界，必须先设计再实现。
- 数据库 migration 属于高风险边界，必须有验证和回滚思路。
- 上传和 AI 可后置，避免拖慢 v0.1 主闭环。
