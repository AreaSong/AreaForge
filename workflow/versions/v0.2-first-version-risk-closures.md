# v0.2 完整第一版高风险闭环

## 计划状态

- 计划状态：**已完成**。完成证据以 `docs/development/docs-100-completion-record.md` 为准；文中残余证据边界（如 release evidence backup hash）以残余风险台账当前状态为准。

## 目标

把当前低风险基础版推进到 `docs/product/feature-scope.md` 中“第一版必须有”的真实可用状态，重点补齐附件、真实 AI、结构化打卡/债务/反假学习/掌握证明这些不能只靠派生规则长期支撑的能力。

本版本是完整第一版的高风险闭环总入口。`v0.3` 是其中 Package B 结构化学习状态的专项执行线；v0.2 不能在 v0.3 对应结构化状态未完成时退出。

当前进度：Package A 附件上传与鉴权访问、Package B 结构化学习状态 Batch 0-6、Package C 真实 AI Provider 第一版、Package D Batch D1-D5、Package E E1-E4、远端 `v0.1.5` 历史 GitHub Release 签名发布和当前 `v0.1.7` 服务器侧 apply/health/extra-smoke 均已完成；后续只保留暂缓项、post-update redacted OPS 证据、release evidence backup hash、生产写入型 smoke/自动策略等可选增强；这些残余证据不阻塞 Package A-E 功能完成，但会阻止长期运营完成声明。

## 范围

- 附件上传、鉴权下载或受控预览。
- AI 鞭策、复盘建议、明日任务建议的真实 provider 接入。
- 结构化学习收口、打卡快照、任务债务事件账本、恢复状态。
- 掌握条件、证据引用和复测记录基础结构。
- 首页、统计、报告和作战地图统一消费结构化状态。
- 对应 API、UI、core 规则和验证文档同步。

## 不包含

- AI 自动生成完整学习计划。
- AI 自动覆盖用户任务、复盘、阶段计划或附件。
- 复杂 PDF 自动解析。
- 多用户系统、复杂权限系统、排名系统。
- 生产发布和备份恢复执行。

## 入口条件

- v0.1 前置主闭环可用，`pnpm check` 通过。
- `docs/development/feature-traceability.md` 已标明第一版缺口。
- `docs/development/high-risk-confirmation-packets.md` 已列出需要确认的执行包。
- 附件、AI 和结构化 migration 都有影响、风险、验证和回滚说明。
- 用户逐项确认对应高风险包后，才能进入实现。
- 若推进结构化学习状态，按 `versions/v0.3-structured-learning-state.md` 和 `tasks/backlog/0015-structured-state-migration.md` 的批次执行。

## 验收标准

- `docs/product/feature-scope.md` 第一版必须项不再停留在“基础版 / 待确认”。
- 附件不进入 `public/`，通过鉴权 API 访问，metadata/hash/URI 与文件本体一致。
- AI 外呼只在 `AI_ENABLED=true` 时发生，输出结构化校验，失败回退本地规则。
- 打卡、任务债务、恢复模式、反假学习和掌握证明有结构化持久化依据。
- 所有写 API 有服务端鉴权，不泄露密钥、session token、完整 prompt 或隐私正文。
- `pnpm check`、相关包测试、Prisma validate、关键 API 烟测和主要页面验证通过。

## 退出条件

- 用户可以连续使用完整第一版完成“计划任务 -> 专注计时 -> 关联大纲 -> 产出笔记/错题 -> 晚间复盘 -> AI 建议 -> 明日调整”闭环。
- 第一版所有高风险确认包都有实施记录、验证记录和残余风险说明。

## 风险

- migration 失败或旧数据映射不完整。
- 上传目录与数据库 metadata 不一致。
- AI key、prompt 或隐私正文泄露。
- 自动建议被误用为自动覆盖用户记录。
